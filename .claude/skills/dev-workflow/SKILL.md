# dev-workflow

要件定義 → 設計 → 実装(Codex) → 評価 → リリース準備、という一連の開発サイクルを1つの流れとして
進めるための手順書。各フェーズの役割分担を守ることが最も重要な原則:

- **要件定義・設計・評価・リリース準備は Claude Code が行う**
- **実装（コーディング）は Codex に委任する。Claude Code自身はプロダクションコードを直接書かない**
  （設計書の作成やテンプレートファイルの作成など、コーディング以外のファイル操作は対象外）
- **SNS投稿素材などの画像生成は既存の `app-release-promo` スキル（Codex CLI経由）に任せる**

フェーズの切れ目ごとにユーザーに進捗と成果物を提示し、次に進んでよいか確認すること。全フェーズを
無言で自動的に走り切らない（要件が誤っていた場合の手戻りコストが大きいため）。

## 事前確認: Codex MCP連携

このワークフローは `mcp__codex__codex` / `mcp__codex__codex-reply` ツールを使う。Codex MCPサーバーは
プロジェクト単位ではなく `claude mcp add codex -s user -- codex mcp-server` で**ユーザースコープ
（マシン全体）に登録済み**なので、`.claude/skills/dev-workflow` を他プロジェクトへコピーしても
このマシン上であれば追加設定なしにそのまま使える（`claude mcp list` で `codex` が
`✔ Connected` になっていれば有効）。

ツールが見当たらない場合、考えられる原因は次のいずれか:
- 別のマシンで初めて使う場合はまだ登録されていない → `claude mcp add codex -s user -- codex mcp-server`
- 登録直後で現在のセッションに読み込まれていない → セッションを再起動する

`mcp__codex__codex` の入出力仕様（実機で確認済み）:

- 入力: `prompt`（必須）, `cwd`, `sandbox`(`read-only` / `workspace-write` / `danger-full-access`),
  `approval-policy`(`untrusted` / `on-request` / `never`), `model`, `config` など
- 出力: `{ "threadId": string, "content": string }`
- 会話を継続する場合は `mcp__codex__codex-reply` に `threadId` と次の `prompt` を渡す
  （差し戻し・追加修正のたびに新規セッションを作り直さず、同じスレッドで会話を続けること）

実装フェーズでは `sandbox: "workspace-write"`、`cwd` はこのリポジトリのルート、
`approval-policy` は `on-request`（危険な操作だけ確認を挟む）を基本にする。

## 進め方

### フェーズ1: 要件定義

1. ユーザーの依頼から機能名・目的を把握する。曖昧な場合はテンプレートの各項目を埋められる程度まで
   質問する（聞きすぎない。「背景」「スコープ」「受け入れ基準」の3つが埋まれば十分に前に進める）。
2. `templates/requirements_template.md` を土台に要件定義書を作成し、
   `docs/dev-workflow/{{機能名}}/requirements.md` に保存する。
3. 受け入れ基準（5章）は特に丁寧に書く。ここが評価フェーズの合格基準になる。
4. ユーザーに要件定義書を提示し、認識齟齬がないか確認してから次へ進む。

### フェーズ2: 設計

1. `templates/design_template.md` を土台に設計書を作成し、
   `docs/dev-workflow/{{機能名}}/design.md` に保存する。
2. 「7. Codexへの実装指示」は特に重要。Codexは設計書自体を読むわけではなく、この指示文（プロンプト）
   だけを渡されて実装するので、対象ファイル・期待する振る舞い・変更してはいけない範囲を具体的に書く。
3. ユーザーに設計書を提示し、実装に進んでよいか確認する。

### フェーズ3: 実装（Codexに委任）

1. 設計書「7. Codexへの実装指示」を `prompt` として `mcp__codex__codex` を呼び出す。
   `cwd` はリポジトリルート、`sandbox: "workspace-write"` を指定する。
2. 返ってきた `content`（Codexの作業サマリ）と `threadId` を保存する（後続の差し戻しで使う）。
3. Claude Codeは実装コードを自分で書かない。ただし、Codexの出力が指示と食い違う場合や、
   明らかに設計の記述が不十分だったと分かった場合は、`mcp__codex__codex-reply` で同じ `threadId` に
   追加指示を送って修正させる（新しいコードを自分で書いて上書きしない）。
4. 実装が一通り終わったら、変更されたファイル一覧を `git status` / `git diff --stat` などで
   確認し、次フェーズに引き継ぐ。

### フェーズ4: 評価（Claude Codeが実施）

`templates/evaluation_report_template.md` を土台に、以下すべてを行い
`docs/dev-workflow/{{機能名}}/evaluation.md` にまとめる。

1. **受け入れ基準との突き合わせ** — 要件定義書の受け入れ基準を1項目ずつ、実際の挙動やコードを見て
   合格/不合格を判定する。
2. **テスト実行・結果確認** — プロジェクトに既存のテストコマンドがあれば実行する。受け入れ基準を
   満たすための新規テストが必要そうなら、その旨を指摘する（テストコード自体の追加はフェーズ3に
   差し戻して実装フェーズで書かせる。Claude Codeがプロダクション/テストコードを直接書かない原則は
   ここでも維持する）。
3. **コードレビュー** — 設計との整合性、可読性、不要な複雑化がないかを確認する。指摘は
   `ファイル:行` で具体的に。
4. **セキュリティレビュー** — 入力値の扱い、権限、シークレットの取り扱いなど、OWASP Top 10的な
   観点で問題がないか確認する。

総合判定が「要修正」の場合は、具体的な修正内容を「6. Codexへの差し戻し内容」に書き、
フェーズ3に戻って `mcp__codex__codex-reply`（同じ `threadId`）で修正を依頼する。
「合格」になるまでフェーズ3⇄4を繰り返す。ループが3回を超えて収束しない場合は、設計そのものに
問題がある可能性が高いので、いったんユーザーに状況を共有し、設計の見直しを提案する。

### フェーズ5: リリース準備（Claude Codeが実施）

1. 変更内容から、CHANGELOG・バージョン番号・READMEなど必要なドキュメント更新を洗い出し、
   Claude Codeが直接更新する（これはコーディングではなくドキュメント/リリース作業なので対象外にしない）。
2. コミットメッセージ・PR説明の下書きを作る（実際のcommit/push/PR作成はユーザーの明示的な許可を得てから）。
3. SNS告知素材（投稿文・バナー・コンセプト画像）が必要な場合は、このスキルの範囲外として
   `app-release-promo` スキル（Skillツールで `app-release-promo` を呼び出す）に委譲する。
   このスキル内で画像生成やCodex CLI (`codex exec`) の直接呼び出しは行わない —
   `app-release-promo` が既にそのためのテンプレート・スクリプトを持っている。

## 出力先ディレクトリ

```
docs/dev-workflow/{{機能名}}/
    ├── requirements.md
    ├── design.md
    └── evaluation.md   # 差し戻しが発生した場合は上書きせず追記していく
```

## トラブルシューティング

| 症状 | 対処 |
|---|---|
| `mcp__codex__codex` ツールが見当たらない | `claude mcp list` で `codex` が接続されているか確認。未登録なら `claude mcp add codex -s user -- codex mcp-server`、登録済みなら再起動していない可能性 |
| Codexが指示と違うものを実装した | 新しいセッションを立てず、`mcp__codex__codex-reply` で同じ `threadId` に修正指示を送る |
| 評価フェーズでの差し戻しが収束しない | 3周目に入ったら設計書の粒度・曖昧さを疑い、ユーザーを交えて設計を見直す |
| セキュリティレビューで懸念が見つかった | 総合判定を「要修正」にし、具体的な脆弱性シナリオとともにフェーズ3へ差し戻す |
