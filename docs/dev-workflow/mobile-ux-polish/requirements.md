# 要件定義書: モバイルUXの改善（A: 入力のはみ出し解消 / C: タップ領域・余白・ナビ微調整）

作成日: 2026-07-22

## 1. 背景・目的

スマホ閲覧時、一部ページで入力ボックスが画面からはみ出す・潰れる問題がある（実機調査で
基礎生活費・固有の支出のテーブル型入力が 320px 幅で右端の数値入力が切れることを確認）。
また全体的にタップ領域・余白・フォントがデスクトップ前提のままの箇所がある。
将来的なスマホアプリ化を見据え、まずは Web のモバイルUXを実用レベルに引き上げる。

本要件では **A（テーブル型入力のモバイル対応）** と **C（タップ領域・余白・フォント・ナビの微調整）** を対象とする。
**B（閲覧用の横長テーブルのモバイル再設計）** と **D（PWA化）** は本要件のスコープ外とし、別途承認を得てから着手する。

## 2. スコープ

### やること
- **A. テーブル型入力のモバイル対応**
  - 対象: 「基礎生活費」テーブル、「夫/妻の固有の支出」の通常項目テーブル・ローンテーブル
    （`components/data-input/data-input-page.tsx` の `LivingExpenseCard` / `SpecificExpenseCard`）
  - 狭幅（〜約640px, Tailwind `sm` 未満）で、横並びの列レイアウトを **1行=1グループの縦積み** に組み替え、
    ラベルと入力ボックスが画面内に収まるようにする（数値入力の見切れ・列見出しの縦積み潰れを解消）。
  - 320px・375px で入力ボックスが完全に表示され、タップ・入力できること。
- **C. タップ領域・余白・フォント・ナビの微調整**
  - モバイル幅で、主要なインタラクティブ要素（追加/削除ボタン、トグル、タブ、ナビ、入力欄）の
    タップ領域を確保する（高さの目安 44px 以上）。
  - 入力欄・ラベルのフォントサイズが小さすぎないこと（本文入力は概ね 14px 以上）。
  - モバイル上部のナビ（画面切替のセレクト）とタブの余白・折返しを整える。

### やらないこと（今回対象外・別途承認後）
- **B**: ライフプラン明細・住宅ローン返済スケジュールなど閲覧用の横長テーブルのモバイル再設計。
  現状の「カード内で横スクロール」は維持する（壊れてはいないため）。
- **D**: PWA化（manifest・アイコン・インストール・standalone表示）。
- ネイティブアプリ化（React Native / Capacitor 等）。

## 3. 機能要件

| # | 要件 | 優先度 |
|---|---|---|
| 1 | 基礎生活費テーブルが狭幅で縦積みに組み替わり、夫/妻の月額入力が見切れず表示・入力できる | 必須 |
| 2 | 固有の支出（通常項目・ローン）テーブルが狭幅で縦積みに組み替わり、月額・払い終わり等の入力が見切れず表示・入力できる | 必須 |
| 3 | 320px・375px 幅で、データ入力の全タブにおいて入力要素がビューポートからはみ出さない | 必須 |
| 4 | 追加/削除ボタン・トグル・タブ等の主要操作要素のタップ領域が概ね44px以上 | 必須 |
| 5 | 入力欄・ラベルのフォントが読みやすいサイズに保たれる | 任意 |
| 6 | モバイル上部ナビ／タブの余白・折返しが整っている | 任意 |
| 7 | デスクトップ幅（`sm` 以上）の見た目・レイアウトは従来どおり（回帰なし） | 必須 |

## 4. 非機能要件

- 既存の計算ロジック・データ構造には一切変更を加えない（本件は表示（CSS/レイアウト）のみ）。
- レスポンシブは Tailwind のブレークポイント（`sm:` など）で行い、モバイル専用の見た目はデフォルト、
  デスクトップは `sm:` 以上で従来の表示に戻す方針とする。
- アクセシビリティ: 既存の `aria-label` / `aria-pressed` などを維持する。

## 5. 受け入れ基準

- [ ] 320px 幅で、データ入力「支出」タブの基礎生活費・固有の支出テーブルの入力ボックスが画面内に完全表示され、はみ出し・見切れがない
- [ ] 375px 幅でも同様にはみ出しがない
- [ ] データ入力の全タブ（基本設定/収入/支出/住宅ローン/ライフイベント）で、320px・375px 時に横方向のページあふれ（documentElement.scrollWidth > clientWidth）が発生しない
- [ ] 縦積みに組み替えた入力行でも、各値のラベルが分かり、値の追加・変更・削除ができる
- [ ] 追加/削除ボタン・トグル・タブなど主要操作要素のタップ高さが概ね44px以上
- [ ] デスクトップ幅（例: 1280px）でのデータ入力画面の見た目が本改修前と実質同じ（列テーブル表示のまま）
- [ ] `npm test` / `tsc --noEmit` / `eslint` がすべて通る（計算ロジック不変のためテスト結果も不変）

## 6. 前提・制約

- 変更は主に `components/data-input/data-input-page.tsx`（`LivingExpenseCard` / `SpecificExpenseCard` /
  各カードのタップ領域・余白）と、必要に応じて `app/page.tsx` のモバイル上部ナビ、
  `components/data-input/fields.tsx`（入力欄の余白・サイズ）に限定する。
- 計算・型・保存ロジック（`engine/`, `lib/storage.ts`）は変更しない。
- B/D 相当の変更（閲覧テーブル再設計・PWA）は行わない。
